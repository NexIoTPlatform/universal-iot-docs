以下为 `奈科斯` 在落地过程中的高频问题与简要答案，便于快速决策与排障。具体指标与企业实践可在后续补充。

###  刚开源，有实际项目运行验证吗？稳定吗？

::: info ```  
本项目19年开始研发，支撑项目200多个，超百万级别设备接入，稳定运行5年！新兵蛋子不代表新出生！
:::

###  好像功能不完整，不成熟？

::: info ```  
本项目的定位是`IoT设备接入`基础设施，完成统一的南北向设备接入，让你专心做业务，为上层应用提供赋能

</br>如果不认可出门右转！
:::
###  支持哪些协议？如何扩展？

::: info ```  
- 开箱支持：MQTT、TCP、HTTP；提供云云对接（天翼物联、OneNET、第三方平台）。开关按需开启
:::

###  部署要求与依赖是什么？

::: info ```  
- 运行环境：JDK 21+
- 基础依赖：数据库（MySQL）、Redis、MQTT(自建或者EMQX)；
:::

### 如何实现高可用与水平扩展？

::: info
- 开源版支持集群  
:::
###  数据库如何选择？是否支持时序优化？

::: info
- 时序/分析：日志已支持 ClickHouse，IoTDb、Mysql、None(不存）,产品级别动态无感切换
:::

###  TCP接入粘包、拆包、16进制、字符串类型、CRC等等要求需要改接入服务的代码吗？

::: info
- 完全不要！不用在核心代码加`if else`你更需要关注对方的设备文档；但是为了更好的你适配设备，可能需要在Java解码类，完全符合`开闭原则`。
:::

###  规则引擎能像nodered一样拖拽吗？

::: info
- rulego已经上线了
:::

### 是否提供 SDK？如何集成？

::: info
- 提供开箱即用的Java SDK
:::

###  你所谓`零代码`，把设备解析外置性能如何？

::: info
- 底层采用ASM字节码技术，实际上还是原声Java，做了大量的优化
:::


### `零代码`，协议解析的性能如何？

- 底层采用ASM字节码技术，实际上还是原生Java，整个平台支持集群的同时，做了大量的优化

- 平台做了大量实际（DB操作）真实的并发验证，保持高性能，稳定运行。在内部单机8核32G,qps500,19万设备，稳定运行5年
